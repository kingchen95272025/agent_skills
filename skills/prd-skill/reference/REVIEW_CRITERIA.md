# PRD评审标准

本文档定义PRD需求评审的评分标准和质量评级机制，确保PRD文档质量的客观评估。

## 评审概述

### 评审目的

- 客观评估PRD文档质量
- 识别需求文档中的问题和风险
- 提供具体的改进建议
- 确保PRD具备可执行性

### 评审原则

- **客观公正**：基于事实和标准，避免主观臆断
- **全面系统**：覆盖需求、设计、业务、文档等多个维度
- **具体可操作**：问题描述清晰，建议切实可行
- **持续改进**：支持多轮优化，逐步提升质量

## 评估维度

PRD评审采用五维度评估体系，总分10分：

| 维度 | 权重 | 说明 |
|------|------|------|
| 需求完整性 | 25% | 功能覆盖度、边界条件、异常处理 |
| 可执行性 | 25% | 开发理解度、资源匹配度、验收标准 |
| 用户价值 | 20% | 需求匹配度、体验设计、价值创造 |
| 业务逻辑 | 20% | 流程合理性、规则完整性、数据一致性 |
| 文档质量 | 10% | 结构清晰度、描述准确性、格式规范性 |

## 详细评分标准

### 1. 需求完整性（25%，满分2.5分）

#### 1.1 功能覆盖度（10%，满分1.0分）

**评分标准**：

| 得分区间 | 评估标准 |
|----------|----------|
| 0.9-1.0 | 所有核心功能完整覆盖，主要业务场景全部描述，用户角色和权限定义清晰 |
| 0.7-0.8 | 核心功能基本覆盖，主要场景有描述，角色权限基本明确 |
| 0.5-0.6 | 核心功能部分遗漏，场景描述不够全面，角色权限定义模糊 |
| 0.3-0.4 | 功能覆盖度较低，重要场景缺失，角色权限未明确 |
| 0.0-0.2 | 功能严重缺失，场景描述不清，无角色权限定义 |

**检查要点**：
- [ ] 产品规划中的所有功能都有详细描述
- [ ] 核心业务场景全部覆盖
- [ ] 所有用户角色的权限都已定义
- [ ] 功能边界清晰明确

#### 1.2 边界条件（8%，满分0.8分）

**评分标准**：

| 得分区间 | 评估标准 |
|----------|----------|
| 0.7-0.8 | 明确定义功能边界，清晰说明什么不做，约束条件完整 |
| 0.5-0.6 | 边界基本明确，约束条件基本完整 |
| 0.3-0.4 | 边界定义模糊，部分约束条件缺失 |
| 0.0-0.2 | 边界不清晰，约束条件严重缺失 |

**检查要点**：
- [ ] 明确说明功能范围和边界
- [ ] 清晰定义什么在范围内，什么不在范围内
- [ ] 说明技术约束、资源约束、时间约束
- [ ] 定义与其他系统的集成边界

#### 1.3 异常处理（7%，满分0.7分）

**评分标准**：

| 得分区间 | 评估标准 |
|----------|----------|
| 0.6-0.7 | 异常场景全面考虑，错误处理规则明确，容错机制完善 |
| 0.4-0.5 | 主要异常场景已考虑，错误处理基本明确 |
| 0.2-0.3 | 部分异常场景缺失，错误处理规则不清晰 |
| 0.0-0.1 | 异常场景考虑不足，缺少错误处理规则 |

**检查要点**：
- [ ] 识别并描述主要异常场景
- [ ] 定义异常情况的处理规则
- [ ] 设计容错和降级机制
- [ ] 说明用户如何从异常中恢复

- [ ] 每个功能都有明确的验收标准

## 质量评级

### 评分计算

综合得分 = 需求完整性得分 + 可执行性得分 + 用户价值得分 + 业务逻辑得分 + 文档质量得分

### 质量分级

| 等级 | 得分范围 | 质量评价 | 建议行动 |
|------|----------|----------|----------|
| 优秀 | ≥8.5 | PRD质量优秀，可直接发布 | 进入发布准备阶段 |
| 良好 | 7.0-8.4 | PRD质量良好，建议优化后发布 | 进入文档优化阶段 |
| 需要优化 | 6.0-6.9 | PRD存在明显问题，需要优化 | 进入文档优化阶段，重点修复主要问题 |
| 必须优化 | <6.0 | PRD质量不达标，必须大幅改进 | 进入文档优化阶段，全面改进 |

### 评级详细说明

#### 优秀（≥8.5分）

**特征**：
- 需求完整，覆盖全面
- 描述清晰，无歧义
- 业务逻辑合理完整
- 用户价值明确
- 文档质量高

**建议**：
- 可以直接进入发布准备
- 制作最终交付版本
- 分发给开发团队

#### 良好（7.0-8.4分）

**特征**：
- 需求基本完整
- 描述基本清晰
- 业务逻辑基本合理
- 有明确的用户价值
- 文档质量良好

**建议**：
- 建议进行一轮优化
- 修复主要问题
- 提升到8.5分以上

#### 需要优化（6.0-6.9分）

**特征**：
- 需求不够完整
- 部分描述不清晰
- 业务逻辑有不合理之处
- 用户价值不够突出
- 文档质量一般

**建议**：
- 必须进行优化
- 重点修复P0和P1问题
- 目标提升到7.5分以上

#### 必须优化（<6.0分）

**特征**：
- 需求缺失较多
- 描述模糊或有歧义
- 业务逻辑存在严重问题
- 用户价值不明确
- 文档质量较差

**建议**：
- 必须大幅改进
- 全面修复各类问题
- 可能需要重新梳理需求和规划

## 评审报告模板

评审报告应包含以下内容：

1. **评审概要**：评审时间、版本、得分、结论
2. **评分详情**：五个维度的详细评分和说明
3. **综合评估**：主要优点、主要问题、优化建议
4. **评审结论**：质量评级和下一步建议

详细模板请参考 [phases/04-review.md](../phases/04-review.md)

## 评审流程

1. **准备评审**：确认PRD文档草稿已完成
2. **逐项评估**：按照评分标准逐项评估
3. **计算得分**：汇总各项得分得出综合得分
4. **编写报告**：撰写详细的评审报告
5. **反馈沟通**：与产品经理沟通评审结果
6. **确定后续**：决定是否进入优化阶段

## 注意事项

1. **客观评审**：基于标准和事实，避免主观判断
2. **全面分析**：不遗漏任何重要问题
3. **具体建议**：提供可操作的改进方案
4. **正面激励**：肯定优点，鼓励改进
5. **持续跟踪**：跟踪优化效果，确保质量提升
