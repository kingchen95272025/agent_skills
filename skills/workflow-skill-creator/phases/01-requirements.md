# 第一阶段：需求分析与角色定位

## 阶段概述

这是Workflow Skill创建的第一个阶段，目标是深度理解用户需求，明确AI助手的角色定位和核心价值主张。

## 触发条件

- 用户提出创建工作流Skill的需求
- 或明确表示要进行需求分析
- 或相关目录不存在任何文档

## 阶段目标

1. 收集完整的基础信息（领域、角色、价值主张、用户画像、输入输出等）
2. 通过深度访谈了解行业特殊性和专业要求
3. 生成完整的需求分析文档
4. 获得用户对需求分析的明确确认

## 具体工作内容

### 1.1 基础信息收集

使用以下问卷引导用户提供核心信息：

#### 基础定位问卷

**A. 领域与角色**

1. **专业领域**：您希望AI助手服务于哪个领域？
   - [ ] 内容创作（写作、编辑、文案）
   - [ ] 产品管理（需求分析、产品设计）
   - [ ] 技术开发（编程、架构、测试）
   - [ ] 设计创意（UI/UX、平面、品牌）
   - [ ] 数据分析（报告、洞察、建模）
   - [ ] 教育培训（课程、教案、评估）
   - [ ] 营销推广（策略、文案、分析）
   - [ ] 其他：___________

2. **角色名称**：如何称呼这个AI助手？
   - 建议：简洁、专业、易记（如Writer、Designer、Analyzer）
   - 格式要求：kebab-case（如content-writer、data-analyzer）

3. **专业身份**：这个助手的专业身份定位是什么？
   - 示例：高级内容策划师、资深产品经理、全栈开发专家

**B. 价值主张**

4. **核心问题**：要解决的主要痛点是什么？
5. **独特价值**：相比其他助手的独特优势是什么？
6. **成功标准**：用户用后会有什么显著改善？

**C. 用户画像**

7. **目标用户**：主要服务对象是谁？
8. **用户特征**：他们的专业水平、工作场景、行为特点？
9. **使用场景**：典型的使用情境和频次？

**D. 输入输出**

10. **输入类型**：用户通常会提供什么样的原始材料？
11. **输出形式**：最终交付什么类型的成果？
12. **质量标准**：输出成果的质量要求和评判标准？

### 1.2 深度访谈

基于用户的回答，进行深度访谈，了解：

- **行业特殊性**：该领域有哪些独特的专业要求？
- **工作流程痛点**：现有流程的复杂度和主要困难？
- **交互方式偏好**：期望的沟通风格和互动方式？
- **成功案例参考**：有没有类似的工具或助手作为参考？
- **特殊需求**：是否有特定的约束、规范或标准？

### 1.3 需求文档生成

创建 `.claude/skill-creator/{skill-name}/requirements.md` 文档，包含以下结构：


## 重要约束

1. **完整性要求**
   - 必须收集所有必要信息才能进入下一阶段
   - 如果用户信息不完整，必须继续提问直到明确
   - 不能假设或猜测用户的需求

2. **确认机制**
   - 必须等待用户的明确认可才能进入下一阶段
   - 如果用户提供反馈，必须修改并再次请求确认
   - 必须继续反馈-修订循环直到获得明确批准

3. **深度理解**
   - 不仅要收集表面信息，还要理解深层需求
   - 要主动识别潜在的问题和机会
   - 要考虑用户可能没有明确提到的重要方面

4. **专业性**
   - 使用专业的产品经理思维分析需求
   - 识别需求中的矛盾和不合理之处
   - 提供专业的建议和优化思路

## 交互示例

### 示例1：完整流程

**AI**: "您好！我看到您想创建一个workflow skill。让我们从需求分析开始。首先，请告诉我：

1. 这个AI助手主要服务于哪个专业领域？
2. 您希望如何称呼这个助手？
3. 它要解决的核心问题是什么？"

**用户**: "我想做一个帮助写技术文档的助手，叫TechWriter，主要解决技术文档写作效率低、质量不稳定的问题。"

**AI**: "很好的想法！让我继续了解更多细节：

1. **目标用户**：主要是谁在使用？是开发者、技术写作人员，还是产品经理？
2. **输入材料**：用户通常会提供什么？是代码、接口文档、产品需求，还是其他？
3. **输出要求**：最终需要什么样的技术文档？API文档、用户手册、还是架构文档？"

**用户**: "主要是开发者使用，他们会提供代码和简单的功能说明，需要生成标准的API文档和用户使用指南。"

**AI**: "明白了。让我确认几个细节：

1. **专业水平**：使用者是初级开发者还是有经验的？
2. **质量标准**：API文档需要遵循特定的规范吗？比如OpenAPI、JSDoc？
3. **工作场景**：是在项目开发过程中持续生成，还是版本发布前集中编写？"

**用户**: "主要是中高级开发者，需要遵循OpenAPI 3.0规范，通常是在迭代开发中持续更新。"

**AI**: "太好了！现在我对需求有了清晰的理解。让我生成一份完整的需求分析文档..."

[生成requirements.md文档]

**AI**: "需求分析完成！我已经整理了完整的需求文档，包括：
- 核心定位：面向中高级开发者的API文档生成助手
- 输入输出：代码+功能说明 → OpenAPI 3.0规范文档
- 工作方式：迭代式、协作式文档生成

请查看是否准确反映了您的需求？如果有需要调整的地方，请告诉我。确认无误后，我们就可以进入流程设计阶段了。"

### 示例2：需求调整

**用户**: "输出不只是API文档，还需要包括示例代码和最佳实践说明。"

**AI**: "好的，我理解了。让我更新需求文档，在输出形式部分增加：
- API文档（OpenAPI 3.0规范）
- 使用示例代码
- 最佳实践说明

这样更完整地反映了输出要求。我马上更新文档..."

[更新requirements.md]

**AI**: "已更新。现在输出规格更全面了。还有其他需要调整的地方吗？"

## 完成标志

当满足以下条件时，第一阶段完成：

1. ✅ 已创建 `requirements.md` 文档
2. ✅ 文档包含所有必要的8个部分
3. ✅ 用户明确表示满意当前需求分析
4. ✅ 没有遗漏或模糊的关键信息

## 自然确认询问

阶段完成时，使用以下方式之一确认：

- "需求分析完成，我已经深度理解了您的{领域}助手需求。是否可以进入流程设计阶段？"
- "需求文档看起来完整了，涵盖了{核心要点}。准备好设计六阶段工作流了吗？"
- "我已经清楚地了解了{Skill名称}的定位和目标。要不要开始设计具体的工作流程？"
